## Project Overview

A chore tracker for families. Kids select themselves via profile image (no login), complete chores, and admins (parents) approve completions and manage payouts.

## Commands

### Backend (Rust)

```bash
cargo run         # Start dev server (localhost:7007)
cargo test        # Run all tests
just test         # Run tests with file-watching (watchexec)
just cover        # Generate lcov coverage report
```

### Frontend (`cd site` first)

```bash
yarn dev          # Dev server (proxies API to localhost:7007)
yarn build        # TypeScript + Vite build → site/build/
yarn test         # Vitest tests
yarn lint         # ESLint + Prettier (auto-fix)
yarn storybook    # Storybook on port 6006
```

### Database

```bash
diesel migration generate <description>   # Create new migration
diesel migration run                      # Apply migrations
diesel migration revert                   # Revert last migration
diesel print-schema > src/schema.rs       # Regenerate schema after migration
```

## Architecture

Full-stack app: Rust/Axum backend with a React/TypeScript frontend. The frontend is built as static files embedded in the Rust binary via `rust-embed`. Diesel migrations run automatically on startup.

### Backend (`/src/`)

- `main.rs` — entry point, sets up server, runs migrations, configures tracing
- `models.rs` — Diesel models and Juniper GraphQL objects
- `graphql.rs` — GraphQL schema and all query/mutation resolvers
- `svc/` — business logic services (one file per entity: `user.rs`, `chore.rs`, `chore_completion.rs`, etc.)
- `api/` — HTTP routes (auth, graphql endpoint, image upload/serving)
- `routes.rs` — Axum routing and middleware (compression, static files)
- `auth.rs` — OIDC authentication for admins
- `schema.rs` — auto-generated by Diesel, do not edit manually

GraphQL is at `/graphql`. REST endpoints are minimal; prefer GraphQL for new features.

### Frontend (`/site/src/`)

- `App.tsx` — Apollo Client setup, routing, lazy-loaded pages
- `components/` — reusable UI components
- `page/` — page-level components (lazy loaded)
- `hooks/` — all GraphQL operations wrapped in custom hooks (Apollo `useQuery`/`useMutation`)
- `queries.ts` — shared GraphQL query strings
- `types/` — TypeScript interfaces

## Key Conventions

**GraphQL naming**: Rust fields are `snake_case`; Juniper auto-converts to `camelCase` for the GraphQL schema. Boolean fields do NOT use `is` prefix — use `active` not `isActive`.

**Frontend GraphQL**: All query/mutation logic belongs in hooks under `site/src/hooks/`. Access query results via the operation name in `data` (e.g., `data.listChores`).

**Rust imports**: Prefer importing at the module level and calling with short paths (e.g., `use std::cmp; cmp::min(...)`) rather than fully qualifying inline.

**Error handling**:
- Backend: use `anyhow::Result`, convert to GraphQL errors with `graphql_translate_anyhow()`
- Frontend: surface errors via `toast.error()`

**Package manager**: Always use `yarn` in the `site/` directory, never `npm`.

## Adding New Features

1. Create Diesel migration (`diesel migration generate`)
2. Update `src/models.rs`
3. Add service logic in `src/svc/{entity}.rs`
4. Add GraphQL resolvers in `src/graphql.rs`
5. Update TypeScript types in `site/src/types/`
6. Add hook in `site/src/hooks/`
7. Create/update React components and pages

## Database Notes

- SQLite at `./db/db.sqlite` (auto-created)
- Models use UUIDs for external references; integer IDs for internal foreign keys
- Foreign keys enabled via pragma
- Always test migrations in both directions: `run` → `revert` → `run`

## Weekly Chore Payment Logic

Weekly chores split payment across assigned days (e.g., $1.50 chore on 3 days = $0.50/day). Required days are stored as a bitmask (1=Mon, 2=Tue, 4=Wed, 8=Thu, 16=Fri, 32=Sat, 64=Sun). See `WEEKLY_CHORE_FIX.md` for the correction mutations if payment amounts need fixing.
